# Qubes deployment helpers

Running the aims teamserver as a persistent service inside a Qubes AppVM (the
`sec` qube), driven from the dev qube (`sec-dev`) with the `aims-install` zsh
function (see `~/.config/zsh/config.d/aims.zsh`).

## Pieces

| Piece | Lives in | Role |
|-------|----------|------|
| `aims-deploy` (zsh) | sec-dev | build → `qvm-copy` → `sec.AimsInstall` → binary into `sec:~/.local/bin/aims` |
| `aims-install` (zsh) | sec-dev | `aims-deploy` **plus** `sec.AimsInstallService` → teamserver as a user service |
| `sec.AimsInstall` (qrexec) | sec | installs the copied binary (pre-existing) |
| `sec.AimsInstallService` (qrexec) | sec | renders + enables the systemd **user** service (this dir) |
| `aims-teamserver.service` | sec (`~/.config/systemd/user/`) | the unit (`contrib/systemd/`, or generated by the qrexec service) |

Why a **user** service: an AppVM's root FS is non-persistent, so a
`/etc/systemd/system` unit would not survive a reboot. `~/.config/systemd/user`
is on the persistent `/home` volume; `loginctl enable-linger` starts it at boot.

## One-time setup (mirrors the existing `sec.AimsInstall`)

1. In `sec`: copy `sec.AimsInstallService` next to `sec.AimsInstall`
   (e.g. `~/.local/etc/qubes-rpc/sec.AimsInstallService`) and `chmod +x` it.
2. In `dom0`: add a policy line (e.g. in `/etc/qubes/policy.d/30-user.policy`):

   ```
   sec.AimsInstallService * sec-dev sec allow
   ```

## Use

```sh
aims-install                 # build, deploy binary, install+enable the service in `sec`
# then, once, in `sec`:
aims teamserver user --system --host 127.0.0.1 --port 31337
```

After that, any `aims <command>` in `sec` auto-detects the system config in
`~/.aims/teamclient/configs/` and runs as a thin client of the local teamserver —
no teamserver or database is constructed client-side.

Change the bind host/port in `sec.AimsInstallService` **and** the `user --system`
command together (they must match).
